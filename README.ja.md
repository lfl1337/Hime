# Hime — 日本語ライトノベル英訳スイート

> **[English README → README.md](README.md)**

Hime は、多段階AIパイプラインを使って日本語ライトノベルを英語に翻訳するローカルファーストのデスクトップアプリです。翻訳はすべてマシン上で完結し、データは外部に送信されません。

**現在のバージョン:** v2.0.0  
**対象ジャンル:** 百合系ライトノベルを中心とした文学翻訳（プロンプトとグロッサリーでジャンル調整可能）

## 主な機能

- **4段階翻訳パイプライン** — 4モデルが並列でドラフトを生成し、辞書アンカーと組み合わせた後、マージ → ポリッシュ → リーダーパネルによるレビューを実施
- **15ペルソナ リーダーパネル** — 小型モデルが仮想ベータリーダーとして最終出力を読み、問題を検出。2段階のリトライループ（軽微修正 / 完全リトライ）で品質を担保
- **既刊ボリュームへの RAG** — sqlite-vec + bge-m3 埋め込みにより、シリーズを通じて用語・人名・敬称の一貫性を維持
- **EPUB インポート / エクスポート** — 日本語 EPUB をドラッグ&ドロップ、章ごとに翻訳し、ebooklib で英語 EPUB としてエクスポート
- **ライブストリーミング** — WebSocket を通じて各ステージのトークン生成をリアルタイム表示
- **トレーニングモニター** — LoRA アダプタをローカルでファインチューニング。損失グラフ、ハードウェアモニタリング、カリキュラム学習に対応
- **データレジストリ** — トレーニングデータソースの一元管理。レビュー済み翻訳 → 次のトレーニングランというフライホイールループに対応
- **オフラインファースト** — EPUBライブラリ、翻訳履歴、グロッサリー、RAGストアはすべてモデルが起動していなくても動作

## アーキテクチャ

```
前処理
  ├── EPUB 抽出
  ├── MeCab 分かち書き
  └── RAG コンテキスト取得（sqlite-vec + bge-m3）

Stage 1 — 並列ドラフト生成（4モデル + 辞書アンカー）
  ├── Qwen2.5-32B               （LoRA、アンカーモデル — 日本語理解力最強）
  ├── TranslateGemma-12B        （LoRA、機械翻訳特化）
  ├── Qwen3.5-9B                （LoRA、高速な最新 Qwen）
  ├── llm-jp-3-7.2b-instruct3   （日本語ファーストの多様性ドラフト、ノートレーニング）
  └── JMdict / MeCab            （アルゴリズム辞書アンカー — 語彙の正規表現）

Stage 2 — コンセンサスマージ
  └── TranslateGemma-27B        （5入力をマージ、JMdict に対してハルシネーションを拒否）

Stage 3 — ポリッシュ
  └── Qwen3-30B-A3B MoE         （軽微な編集、文体整合 — ゼロショット）

Stage 4 — リーダーパネル + アグリゲータ
  ├── 15× Qwen3.5-2B            （システムプロンプトによるペルソナベースリーダー）
  └── LFM2-24B-A2B              （アグリゲータ：okay / fix_pass / full_retry 判定を出力）

後処理
  └── ebooklib による EPUB エクスポート
```

Stage 1 モデルはすべて1枚の GPU で逐次ロードされ、VRAM を積極的に管理します。RTX 5090（32 GB VRAM）向けに設計されていますが、それ以下のカードでは CPU オフロードにフォールバックします。

## 動作環境

- Windows 10/11（メインターゲット）、Linux はサポート対象
- NVIDIA GPU 24 GB VRAM 以上推奨（Stage 1 並列実行には 32 GB）
- [Conda](https://docs.conda.io/)（`hime` 環境、トレーニング/ML 用）
- [`uv`](https://github.com/astral-sh/uv)（バックエンド Python 環境）
- Node.js 22 以上、npm
- Rust ツールチェーン（Tauri ビルド用）
- （オプション）Ollama、llama.cpp、または vllm（GGUF 推論用）

## セットアップ

### 1. クローンと設定

```bash
git clone https://github.com/lfl1337/Hime.git Hime
cd Hime
cp .env.example .env
# .env を編集してパスを設定（コメントアウトのままにすれば自動検出）
```

### 2. バックエンド（FastAPI + uv）

```bash
cd app/backend
uv sync
uv run python run.py
```

バックエンドは `127.0.0.1:18420` にのみバインドされ、外部ネットワークには公開されません。

### 3. フロントエンド（Tauri + React）

```bash
cd app/frontend
npm install
npm run tauri dev    # または: npm run vite（ブラウザのみの開発モード）
```

### 4. トレーニング環境（Conda）

```bash
conda activate hime
# ML スタックを確認:
python -c "import unsloth, transformers, trl, peft; print('OK')"
```

### 5. モデルのダウンロード

デフォルトでは `modelle/` 以下にモデルファイルを配置します。`huggingface-cli` でダウンロード：

```bash
# bge-m3 埋め込みモデル（RAG に必要）
huggingface-cli download BAAI/bge-m3 --local-dir modelle/embeddings/bge-m3

# Stage 1 ベースモデル（大容量 — 完全なリストは pipeline_v2.py を参照）
huggingface-cli download Qwen/Qwen2.5-32B-Instruct --local-dir modelle/qwen2.5-32b
```

すべてのモデル ID と期待されるパスは `app/backend/app/config/pipeline_v2.py` を参照してください。

## 初回セットアップ：RAG の構築

RAGベクターストアはリポジトリに含まれていません。EPUBライブラリからローカルで生成されるもので、翻訳するシリーズに固有のデータです。クローン直後はストアが空の状態です：

**Step 1 — EPUB のインポート**

アプリを起動し、**ライブラリ** タブを開いて、日本語 EPUB ファイルをドラッグ&ドロップします。Hime はすべての章を解析し、ローカルの SQLite データベースに保存します。

**Step 2 — RAG インデックスの構築**

**設定 → RAG** に移動し、**ライブラリを再インデックス** をクリックします。bge-m3 埋め込みモデルがインポートされたすべての段落に対して実行され、結果が `data/rag/` に sqlite-vec ストアとして書き込まれます。

インデックス作成時間はライブラリのサイズと GPU の速度に依存します。10 巻シリーズ（約 3000 段落）で RTX 3080 使用時に約 5〜10 分かかります。

**Step 3 — 動作確認**

インデックス作成が完了したら、段落を翻訳してみてください。パイプラインログに `[RAG] found N context chunks` が前処理ステージで表示されれば成功です。`0 chunks` と表示される場合は、`HIME_EMBEDDINGS_DIR` が bge-m3 モデルを指しているか、`HIME_RAG_DIR` が書き込み可能かを確認してください。

> Hime Vault（`Hime-vault/`）は、同じ RAG コンテンツを人間が閲覧するための Obsidian ベースのオーバーレイです。同じインデクサーによって再構築されるため、gitignore されています。パイプラインの動作に Obsidian は必要ありません。

## パス設定

すべてのパスは環境変数から導出されます。`.env` で設定してください：

| 変数                  | デフォルト                    | 用途                                |
|-----------------------|-------------------------------|------------------------------------|
| `HIME_PROJECT_ROOT`   | 自動検出                      | ベースディレクトリ                  |
| `HIME_DATA_DIR`       | `${ROOT}/data`                | EPUB、トレーニングデータ、RAGストア |
| `HIME_MODELS_DIR`     | `${ROOT}/modelle`             | LoRA アダプタ、GGUF、Safetensors   |
| `HIME_EMBEDDINGS_DIR` | `${ROOT}/modelle/embeddings`  | RAG 用 bge-m3 埋め込みモデル        |
| `HIME_LOGS_DIR`       | `${ROOT}/app/backend/logs`    | バックエンド、トレーニング、監査ログ |
| `HIME_DRY_RUN`        | 未設定                        | `1` に設定すると実モデルなしでパイプラインを実行（テストモード）|

## セキュリティ

- バックエンドは **`127.0.0.1` のみ** にバインド — ネットワーク非公開
- 11 種以上のプロンプトインジェクションパターンによる入力サニタイズ
- EPUB インポートパスへのパストラバーサル防御
- 追記専用 JSON-Lines 監査ログ（`logs/audit.log`）
- 翻訳・パイプラインエンドポイントへのレートリミット
- 翻訳中の外部 API 呼び出しなし — すべてローカル実行
- OpenAI SDK はローカル推論サーバー（llama.cpp、vllm、Ollama — すべて `127.0.0.1`）のクライアントとしてのみ使用

> **API 認証について：** Tauri シェルと FastAPI バックエンド間に意図的に認証トークンはありません。両方とも同じマシン上で動作し、バックエンドはループバック以外からの接続を拒否します。バックエンドをスタンドアロンで実行する場合（Tauri なし）、マシン上の任意のプロセスがアクセス可能であることに注意してください。

## 既知の制限事項

- 初回モデルダウンロードは大容量（Qwen3-30B-A3B + bge-m3 で約 17 GB、フル v2 モデルセットはさらに多い）
- 約 300 ページのボリュームを翻訳するのに RTX 5090 で約 20〜30 時間（リトライループの頻度による）
- 現在の Qwen2.5-32B LoRA は評価セットで軽微なオーバーフィッティングが見られる。カリキュラム学習は実装済みだがデータセットごとの調整が必要な場合がある
- リーダーパネルは 1 ボリュームあたり約 7 時間を追加。ペルソナ数の削減や出力の短縮が最も簡単な最適化手段
