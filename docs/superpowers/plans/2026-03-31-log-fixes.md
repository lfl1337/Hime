Issue 5 (identifier) requires a tauri.conf.json rename and a Rust data-migration update.

**Tech Stack:** Vite 8 (Node.js config), Rust / Tauri 2, TypeScript

> **Pre-condition verified:** `config.py` already has `port: int = 18420` (changed in the previous session). No change needed there.

## Task 1: Fix Vite Proxy Port Detection

**Root cause:** `readBackendPort()` in `vite.config.ts` reads `../backend/.runtime_port` as plain text. That file was renamed to `hime-backend.lock` (JSON `{"port":N,"pid":N}`) in the previous session. Because the file doesn't exist, Vite falls back to port 8004, which is a different local service — causing all proxied requests to fail with 401 (wrong service auth) or 404 (endpoints don't exist on that service).

**Files:**
- Modify: `app/frontend/vite.config.ts`

- [ ] **Step 1: Replace `readBackendPort()` with JSON-aware version**

In `app/frontend/vite.config.ts`, replace the entire `readBackendPort` function:

```typescript
// Read the backend port from .runtime_port at Vite startup.
// Node.js can read the file directly — no Tauri plugin or CORS involved.
function readBackendPort(): number {
  const filePath = path.resolve(__dirname, '../backend/.runtime_port')
  try {
    const content = readFileSync(filePath, 'utf8')
    const port = parseInt(content.trim(), 10)
    if (!isNaN(port)) {
      console.log(`[vite] Backend port: ${port} (from ${filePath})`)
      return port
    }
    console.error(`[vite] .runtime_port is not a number: "${content.trim()}"`)
  } catch {
    console.warn(`[vite] .runtime_port not found at ${path.resolve(__dirname, '../backend/.runtime_port')} — defaulting to 8004`)
  }
  return 8004
}
```

with:

```typescript
// Read the backend port from hime-backend.lock at Vite startup.
// Node.js can read the file directly — no Tauri plugin or CORS involved.
// The lock file is JSON: {"port": N, "pid": N} (written by run.py on startup).
function readBackendPort(): number {
  const filePath = path.resolve(__dirname, '../backend/hime-backend.lock')
  try {
    const content = readFileSync(filePath, 'utf8')
    const lock = JSON.parse(content) as { port?: number; pid?: number }
    if (typeof lock.port === 'number' && !isNaN(lock.port)) {
      console.log(`[vite] Backend port: ${lock.port} (pid ${lock.pid ?? '?'}) from ${filePath}`)
      return lock.port
    }
    console.error(`[vite] hime-backend.lock has no valid port field: ${content.trim()}`)
  } catch {
    console.warn(`[vite] hime-backend.lock not found at ${filePath} — defaulting to 18420`)
  }
  return 18420
}
```

Key changes:
- Filename: `../backend/.runtime_port` → `../backend/hime-backend.lock`
- Parsing: `parseInt(content.trim())` → `JSON.parse(content).port`
- Fallback: `8004` → `18420`
- Warning/error messages updated

## Task 2: Startup Log Spam Guard

**Root cause:** The message `dev mode — sidecar skipped, connecting to external backend` accumulates in `backend.log` on every `tauri dev` restart (the log is opened in append mode). Additionally, in edge cases where Tauri triggers the setup closure more than once in the same process, all setup work would be duplicated. A `static AtomicBool` prevents both problems.

**Files:**
- Modify: `app/frontend/src-tauri/src/lib.rs`

- [ ] **Step 1: Add `AtomicBool` import and static guard**

At the top of `lib.rs`, the current imports are:
```rust
use std::fs::{self, OpenOptions};
use std::io::Write;
use std::sync::{Arc, Mutex};
```

Add the atomic import:
```rust
use std::fs::{self, OpenOptions};
use std::io::Write;
use std::sync::{Arc, Mutex};
use std::sync::atomic::{AtomicBool, Ordering};
```

Then, directly above `pub fn run()`, add the static guard:
```rust
/// Prevents setup() from running more than once per process.
/// Guards against Tauri calling setup() on repeated HMR-triggered reloads
/// and prevents log file spam on `tauri dev` restarts.
static SETUP_GUARD: AtomicBool = AtomicBool::new(false);
```

- [ ] **Step 2: Add the guard check at the start of `setup`**

Inside the `.setup(|app| {` closure, insert as the very first line (before the `app_data_dir` resolve):

```rust
        .setup(|app| {
            // Skip if already initialized in this process lifetime.
            if SETUP_GUARD.swap(true, Ordering::SeqCst) {
                return Ok(());
            }

            // ── Resolve runtime paths ──────────────────────────────────────
            let app_data_dir = app
```

## Task 3: Fix App Identifier to `dev.Ninym.hime`

**Why:** The correct identifier is `dev.Ninym.hime` (Ninym = the project owner namespace). `dev.lfl.hime` was set in the previous session with the wrong owner prefix. Changing the identifier moves the AppData path from `%APPDATA%\dev.lfl.hime` to `%APPDATA%\dev.Ninym.hime`. The Rust migration block must handle both old paths.

**Files:**
- Modify: `app/frontend/src-tauri/tauri.conf.json`
- Modify: `app/frontend/src-tauri/src/lib.rs`
- Modify: `app/frontend/src/api/client.ts`

- [ ] **Step 1: Update identifier in `tauri.conf.json`**

Change line 3:

to:
```json

```

- [ ] **Step 2: Update the migration block in `lib.rs`**

The current migration block (inside `#[cfg(target_os = "windows")]`, before `let logs_dir = ...`) is:

```rust
            #[cfg(target_os = "windows")]
            {
                let old_dir = std::path::PathBuf::from(
                    std::env::var("APPDATA").unwrap_or_default(),
                )
                .join("dev.hime.app");
                if old_dir.exists() && !app_data_dir.exists() {
                    match fs::rename(&old_dir, &app_data_dir) {
                        Ok(()) => {
                            eprintln!(
                                "[hime] migrated data dir: dev.hime.app → dev.lfl.hime"
                            );
                        }
                        Err(e) => {
                            eprintln!("[hime] migration failed (non-fatal): {e}");
                            // App will start fresh in the new location
                        }
                    }
                }
            }
```

Replace it with a two-step migration that covers both old paths:

```rust
            #[cfg(target_os = "windows")]
            {
                let appdata = std::env::var("APPDATA").unwrap_or_default();

                // Step 1: migrate dev.lfl.hime → dev.Ninym.hime (identifier rename)
                let from_lfl = std::path::PathBuf::from(&appdata).join("dev.lfl.hime");
                if from_lfl.exists() && !app_data_dir.exists() {
                    match fs::rename(&from_lfl, &app_data_dir) {
                        Ok(()) => eprintln!(
                            "[hime] migrated data dir: dev.lfl.hime → dev.Ninym.hime"
                        ),
                        Err(e) => eprintln!("[hime] migration failed (non-fatal): {e}"),
                    }
                }

                // Step 2: migrate dev.hime.app → dev.Ninym.hime (original rename, first-run)
                let from_hime = std::path::PathBuf::from(&appdata).join("dev.hime.app");
                if from_hime.exists() && !app_data_dir.exists() {
                    match fs::rename(&from_hime, &app_data_dir) {
                        Ok(()) => eprintln!(
                            "[hime] migrated data dir: dev.hime.app → dev.Ninym.hime"
                        ),
                        Err(e) => eprintln!("[hime] migration failed (non-fatal): {e}"),
                    }
                }
            }
```

Note: `app_data_dir` now resolves to `%APPDATA%\dev.Ninym.hime` because Tauri uses the new identifier. The two migration steps are ordered `dev.lfl.hime` first (most recent data) then `dev.hime.app` (original, for users who never ran the `dev.lfl.hime` version).
